# 阶段 6：验证

分为**静态检查**和**运行时验证**两层，缺一不可。

## 验证策略选择

根据阶段 1 的分类，选择对应的验证重点：

| 分类 | 主要验证 | 次要验证 | 可跳过 |
|------|---------|---------|--------|
| **UI** | 交互验证 + 旅程连续性 | 静态检查 | 运行时 API 验证 |
| **样式** | 视觉一致性（在实现阶段已循环验证） | 静态检查 | 运行时验证 + 旅程验证 |
| **数据** | migration 验证 + 数据完整性 | 现有 API 回归 | UI 验证 |
| **API** | curl 调用验证 + 响应格式 | 静态检查 | UI 验证（除非前端也改了） |
| **逻辑** | 单元测试 + 集成测试 | 静态检查 | UI 验证 |
| **全栈** | 全部验证 | 无 | 无 |

样式类验证特殊说明：视觉循环在阶段 5 已经完成了迭代验证，阶段 6 只需做静态检查（tsc）确认没有语法错误即可。

## 6a. 静态检查（必须通过，但不能当作唯一验证）【适用分类：全部】

1. **类型检查**：`npx tsc --noEmit`（前端）
2. **单元测试**：`cd backend && uv run pytest tests/ -v`（如有后端改动）

> **静态检查的局限性**：tsc 只验证类型，pytest 通常用 mock 数据。它们**无法验证运行时行为**（如 API 是否真的按颜色过滤了结果）。静态检查全部通过 ≠ bug 已修复。

## 6b. 运行时验证（核心验证步骤）【适用分类：API、数据、逻辑、全栈】

**有后端 API 改动时**，必须用 curl/httpie 直接调用运行中的 API，验证返回结果：

```bash
# 示例：验证颜色过滤是否生效
curl -s -X POST http://localhost:8002/v1/routes/detect-holds \
  -H "Content-Type: application/json" \
  -d '{"background_url": "/v1/analysis/.../background", "target_color": "red"}' \
  | python -m json.tool | head -20
# 预期：只返回 red 颜色的 holds，不应有 blue/green 等
```

**规则**：
- 构造能**直接触发 bug 场景**的请求（不是随便调个 /health）
- 检查响应中的**关键字段**是否符合预期（不只是 200 OK）
- 如果 API 需要先有数据（如需要先上传视频），用已有的 analysis_id 测试
- 无法从沙盒访问运行中服务时，提供完整的 curl 命令让用户手动验证

**有前端改动时**：
- 描述用户操作步骤，让用户在设备上验证
- 或用 `uv run python -c "..."` 直接调用被修改的函数验证输出

## 6c. 逻辑验证【适用分类：UI、逻辑、全栈】

- **交互验证**（如涉及 UI）：逐步检查用户操作流程是否符合阶段 4 设计
- **根因验证**：检查改动是否真的修复了根因（不只是消除表面症状）

## 6d. 用户旅程连续性验证【适用分类：UI、数据、API、全栈；样式类可跳过】

回到阶段 1 定位的用户旅程上下文，检查（详见 `checklists/journey-continuity.md`）：

- **上游衔接**：从上一步过来的数据/状态，改动后还能正确接收吗？
- **下游传递**：这一步的输出，下一步还能正常消费吗？
- **旁路影响**：如果旅程中有分支路径（如"审核通过"vs"审核拒绝"），两条路径都还正常吗？

**常见断链模式：**
| 改了什么 | 容易断在哪 |
|---------|-----------|
| 修改了数据结构/字段 | 下游页面读不到改后的字段名 |
| 修改了状态流转逻辑 | 下游依赖的状态值变了 |
| 修改了页面路由/入口 | 上游的跳转链接失效 |
| 修改了 API 返回格式 | 所有消费这个 API 的页面 |

如果改动是纯后台/纯算法，不影响用户可见行为，可以跳过此步并说明原因。

**验证失败** → 回到阶段 5 修复，不要跳过。

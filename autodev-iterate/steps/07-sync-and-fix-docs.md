# 阶段 7：同步文档 + 设计修正（强制）

改完代码后，**必须**执行两项文档操作：

## 7a. 设计漂移修正（如有偏离）

检查本次改动是否偏离了设计文档中的决策。如果偏离了，**主动修正设计文档**而不是留着过时的文档给下次迭代埋坑。

**检查清单：**

| 改了什么 | 检查哪个文档 | 如何修正 |
|---------|------------|---------|
| 换了技术方案/库 | design.md 技术选型表 | 更新选型条目，标注修正原因和日期 |
| 改了 API 接口签名 | api.md 端点定义 | 更新请求/响应格式 |
| 改了数据模型字段 | api.md / design.md 数据模型 | 更新字段定义 |
| 改了页面结构/路由 | ui.md 页面清单 | 更新页面描述和布局 |
| 改了组件职责划分 | design.md 架构组件 | 更新组件描述 |

**修正格式**：在被修正的位置添加变更标注，保留历史记录：

```markdown
~~原方案：使用 Library A v2.0~~
**修正（YYYY-MM-DD）**：改用 Library B v3.1，原因：Library A 在 XX 场景下有 YY 问题。
```

**连带更新**：如果项目存在 `*-index.md` 或 `*-rules.md`，且修正内容涉及其中的摘要信息，必须同步更新这两个文件。index.md 和 rules.md 是设计文档的精华摘要，必须与源文档保持一致。

**不修正的情况**：如果改动完全在设计文档的框架内（如修了一个 bug 但没改技术方案），跳过 7a，直接进入 7b。但必须说明"本次改动未偏离设计文档"。

## 7b. 状态同步

调用 `/autodev-sync` 更新设计文档中的功能状态标记。

**不可跳过**：代码改了但文档没更新 = 给下次迭代埋坑（阶段 1 会读到过时信息）。

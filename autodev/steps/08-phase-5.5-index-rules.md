# 步骤 8：阶段 5.5 — 生成项目索引和编码规则

**目标**：从设计文档中提取精炼的索引和规则文档，作为开发阶段的高效上下文载体。

**为什么需要这一步**：开发 subagent 不应加载全部设计文档（容易在长上下文中丢失关键约束）。索引文档和规则文档是设计文档的"精华摘要"，确保 subagent 在有限 token 内获得最关键的信息。

## 产出 1：`*-index.md`（< 100 行）— 项目全景索引

内容结构：
1. **项目概述**（1-3 句）：做什么、为谁做、核心价值
2. **文档清单**：每个设计文档一行摘要 + 文件路径
3. **关键设计决策**：技术栈、架构模式、部署方案（从 design.md 提取，每项一行）
4. **能力-组件映射表**：从 design.md 的映射表精简而来
5. **MVP 功能清单**：从 ideation.md 提取，仅编号+名称+当前状态

## 产出 2：`*-rules.md`（< 80 行）— 编码规则手册

内容结构：
1. **技术栈约束**：框架版本、核心依赖版本号（从 design.md 技术选型表提取）
2. **编码约定**：命名规范、文件组织、组件拆分规则（从 design.md 提取或根据技术栈生成）
3. **API 约定**：认证方式、错误响应格式、分页规范（从 api.md 提取）
4. **质量红线**（精炼版）：
   - 禁止占位：无 TODO/pass/空函数体/硬编码空值
   - 禁止 Mock：无 mock/dummy/fake 数据替代真实调用
   - 禁止降阶：按设计文档方案实现，不替换为更简单的方案
   - 版本正确：依赖版本与技术选型表一致
5. **禁止模式**：从各设计文档中提取的具体 don'ts

## 生成规则
- 从已有的 5 个设计文档中**提取**信息，不要凭空编造
- 严格控制行数：index < 100 行，rules < 80 行。超出时优先删除低优先级信息
- 如果设计文档中某个维度没有明确规定（如没有编码约定），该节留空并标注"待补充"

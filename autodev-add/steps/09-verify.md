# 步骤 9：验证

**目标**：验证新功能可用，且不破坏已有功能。

## 执行

### 9a. 新功能验证

1. **静态检查**：`npx tsc --noEmit`（或对应的类型检查）
2. **运行时验证**：
   - 新增的 API 端点用 curl 实际调用，检查响应
   - 新增的页面在浏览器中验证可用性
3. **红线扫描**：参考 `autodev-shared/checklists/quality-redlines.md`

### 9b. 回归验证（关键）

1. **现有测试**：运行项目的测试套件，确认全部通过
2. **修改过的现有文件**：逐一检查改动是否影响了原有行为
3. **数据模型变更影响**：
   - 如果加了新字段/新表，现有的查询和端点还正常吗？
   - migration 是否是非破坏性的（additive only）？
4. **导航和路由**：新增的页面/路由没有和已有的冲突

### 9c. 验证失败处理

- 新功能有问题 → 回到步骤 8 修复
- 破坏了已有功能 → 优先修复回归问题，再继续新功能
